# Payload CMS + Drizzle Migration Workflow

Manage database schema migrations for the FRAS Canada project using Payload CMS 3.x with the PostgreSQL adapter (Drizzle ORM under the hood).

## Architecture

- **Dev mode (`push: true`):** Schema changes auto-sync to the database on `npm run dev`. No migration files needed during development. This is configured in `src/payload.config.ts` via `push: process.env.NODE_ENV !== 'production'`.
- **Production (`push: false`):** Schema changes require explicit migration files. Migrations are SQL files generated by Drizzle and tracked in `src/migrations/`.
- **Migration state** is tracked in a `payload_migrations` table in the database.

## Commands

| Command | Script | What It Does |
|---------|--------|-------------|
| `npm run migrate:create` | `payload migrate:create` | Generate a new migration file from current schema diff |
| `npm run migrate:create -- add-board-field` | `payload migrate:create add-board-field` | Generate with a descriptive name |
| `npm run migrate` | `payload migrate` | Run all pending migrations |
| `npm run migrate:status` | `payload migrate:status` | Show which migrations have run vs pending |
| `npm run migrate:fresh` | `payload migrate:fresh` | **DESTRUCTIVE** — Drop everything, re-run all migrations |
| `npm run migrate:reset` | `payload migrate:reset` | Roll back all migrations (leaves DB empty) |
| `npm run migrate:refresh` | `payload migrate:refresh` | Roll back all, then re-run all |

## Workflow: Adding/Changing a Collection Field

### During Development (push: true)

1. Edit the collection file in `src/collections/`
2. Run `npm run dev` — Drizzle auto-pushes the schema change
3. Verify in Payload admin (`/admin`)
4. Done — no migration file needed yet

### Before Deploying to Production

1. Ensure your collection changes are finalized
2. Generate migration: `npm run migrate:create -- descriptive-name`
3. Review the generated SQL in `src/migrations/`
4. Test locally: `npm run migrate`
5. Check status: `npm run migrate:status`
6. Commit the migration file alongside the collection changes

### Production Deployment

Payload runs pending migrations automatically on server startup when `prodMigrations` is configured. Alternatively, run `npm run migrate` as a build step.

To configure auto-migration on startup (when ready for production):

```typescript
// In payload.config.ts:
import { migrations } from './migrations'

db: postgresAdapter({
  pool: { connectionString: process.env.DATABASE_URI || '' },
  push: false,
  prodMigrations: migrations,
}),
```

## Troubleshooting

### "relation already exists" error
The schema is out of sync. Options:
1. `npm run migrate:fresh` (dev only — drops all data)
2. Manually fix the migration SQL
3. Delete the `payload_migrations` row for the conflicting migration

### "migration failed" error
1. Check `npm run migrate:status` to see which migration failed
2. Read the generated SQL in `src/migrations/` for syntax issues
3. Check PostgreSQL logs: `docker compose logs postgres`
4. Fix the issue, then re-run `npm run migrate`

### Schema drift between dev push and migration files
If you've been using `push: true` and need to generate your first migration:
1. `npm run migrate:fresh` to start clean
2. `npm run migrate:create -- initial` to capture current schema
3. `npm run migrate` to apply it

### Reset everything (nuclear option)
```bash
docker compose down -v          # Delete database volume
docker compose up -d            # Fresh PostgreSQL
npm run dev                     # push: true recreates everything
```

## Rules

- **Never run `migrate:fresh` or `migrate:reset` against production** — these are destructive
- **Always name your migrations** — `migrate:create -- add-event-publishedDate` not just `migrate:create`
- **Commit migration files with the collection changes** — they should be in the same PR
- **Test migrations locally before deploying** — run against a fresh DB copy
- **One migration per logical change** — don't bundle unrelated schema changes
